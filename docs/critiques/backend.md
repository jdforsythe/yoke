# Backend Critique — plan-draft2.md

## Accept (already correct)
- Files-as-contract between phases (Core Design Principles #1) — dodges the hardest coordination problems and gives us durable artifacts to drive the state machine from. This is the right call for a harness that can crash and resume.
- Hooks own quality gates, not the harness (Core Design Principles #2, Hooks Integration) — right split. The harness would otherwise have to maintain a parallel truth about "did tests pass" and it would drift.
- SQLite via better-sqlite3 with synchronous API (Tech Stack) — correct for a single-writer orchestrator. Sync API removes an entire class of race conditions in the state machine and is a genuine fit here.
- Per-workflow worktrees (Worktree Management) — necessary for parallel workflows, and also the only clean way to do `-c` resume on a specific session tree.
- "No arbitrary timeouts" (Process Management § No Arbitrary Timeouts) — correct instinct. A wall-clock timeout on a long build is how you corrupt a worktree mid-write. Heartbeat on stdout activity is the right signal.
- Capturing raw stream-json to disk (`retain_stream_json: true` in logging config) — the right default. When something breaks mid-run, the raw transcript is the only source of truth.
- Planner defines `review_criteria` alongside `acceptance_criteria` (File Contract § features.json) — anti-rubber-stamp mechanism that doesn't require harness-side cleverness.
- Optional pre-step usage gate is opt-in (Usage Tracking § Optional Pre-Step Usage Check) — keeping this out of the hot path is correct; rate-limit management is a swamp.

## Challenge (propose revision)

### stream-json parsing assumes line-delimited JSON without saying so
**Issue:** Process Management § Streaming shows `child.stdout.on('data', ...)` with a comment "Parse stream-json chunks." Node's `'data'` event fires on arbitrary byte boundaries — a single JSON event can be split across two `data` chunks, and two events can arrive in one chunk. The plan never specifies the framing (NDJSON? length-prefixed? concatenated objects?) or the buffering strategy. "Parse each chunk" as written will break on the first large tool_use event that straddles a read boundary.
**Impact:** Silent data loss in the UI stream, corrupted token-usage accounting, and crash-on-malformed-JSON the first time an event exceeds the pipe buffer (~64KB on macOS). This will not show up in dev and will show up in production the moment an agent pastes a large file into a tool call.
**Proposed change:** Commit to Claude Code's stream-json being NDJSON (needs verification — see Questions), and specify a line-buffered reader: accumulate bytes in a `Buffer`, split on `\n`, parse complete lines, carry the trailing partial line forward. Use `readline.createInterface({ input: child.stdout, crlfDelay: Infinity })` or a hand-rolled splitter — do NOT JSON.parse raw `data` chunks. Also specify behavior for: (a) a line that fails JSON.parse (log + skip, don't crash the pipeline), (b) a line exceeding a sanity cap (e.g., 16MB — truncate and mark the session tainted), (c) stderr interleaving (stderr is a separate stream, never mixed into the JSON parser).

### "Pipe assembled prompt via stdin" ignores EPIPE and backpressure
**Issue:** Process Management § Spawning shows `promptStream.pipe(child.stdin)` with no error handling. If jig/claude exits before consuming the whole prompt (fast error path, auth failure, profile-not-found, rate limit at startup), the write side gets EPIPE and crashes the harness unless explicitly handled. Additionally, large prompts (multi-MB features.json + architecture.md + progress.md + recent diff) can hit the OS pipe buffer and block writes; if the child hasn't started draining yet the harness deadlocks waiting for `drain`.
**Impact:** Harness crashes or hangs on the first unlucky phase start. Hard to reproduce, hard to debug.
**Proposed change:** (1) Attach `child.stdin.on('error', ...)` before piping and treat EPIPE as "child died during prompt send — fail the phase, capture stderr, don't crash." (2) Don't stream the prompt from a file — read it into memory (we already have the whole thing) and `child.stdin.end(promptBuffer)` in one call. (3) Document the prompt size ceiling and validate at assembly time (reject prompts > N MB with a clear error rather than deadlocking). (4) Handle `child.on('spawn', ...)` vs `child.on('error', ...)` — ENOENT on jig-not-found arrives as an `error` event on the ChildProcess, not stderr.

### SIGTERM + "wait briefly" is not a process lifecycle
**Issue:** Crash Recovery § On Harness Exit says "Send SIGTERM to all child agent processes. Wait briefly for graceful shutdown. Exit." This is hand-wavy in three ways: (1) no timeout value, (2) no SIGKILL escalation, (3) no handling of grandchildren. jig spawns claude, claude spawns bash for tool calls, bash runs pnpm, pnpm spawns node. Sending SIGTERM to the jig pid does not kill the tree. Node's `child_process` does not create a process group unless you ask it to, so `kill -TERM -pid` won't work either.
**Impact:** On Ctrl-C you get orphaned `claude`, `pnpm install`, `vitest` processes that keep running, keep writing to the worktree, keep consuming tokens, and race with the next harness restart which thinks the workflow was cleanly interrupted. This is the classic "why is my laptop fan on after I killed the harness" bug.
**Proposed change:** Spawn children with `detached: true` and a new session (`setsid`) so each child gets its own process group. On shutdown: `process.kill(-child.pid, 'SIGTERM')`, set a timer (concrete: 10s), then `process.kill(-child.pid, 'SIGKILL')`. Also handle the reverse case — if the harness itself is SIGKILLed, the next startup must detect stale processes. Store `{pid, pgid, started_at}` in the `sessions` table and on restart, probe each (kill(pid, 0)) and reap/mark them cancelled before attempting `-c` resume. Add a `harness doctor` command that lists dangling processes.

### Claude Code `-c` semantics are load-bearing but unverified
**Issue:** The retry loop (Phase 2 § Retry loop), crash recovery (Crash Recovery § On Harness Restart), and the `retry_mode: "continue"` config all assume `claude -c` will reliably resume the previous session in the worktree with the previous context plus new input. The plan never specifies: does `-c` work after an abnormal exit? Does it work after the harness restarts (i.e., across Node processes)? Does it work if another `claude` invocation happened in the same directory in between? Does it work across `jig run <profile>` boundaries, or is session state tied to the exact CLI invocation shape? Does `-c` accept a new `-p` prompt or does it ignore stdin?
**Impact:** If `-c` doesn't do what the plan thinks, both the retry loop and crash recovery collapse. The retry loop silently starts a fresh session that doesn't know why it's being invoked; the crash recovery resumes the wrong session or none at all. This is the single biggest unverified assumption in the plan.
**Proposed change:** Before writing the retry loop, run empirical tests (document in a `docs/research/continue-semantics.md`): (1) happy-path resume after clean exit, (2) resume after SIGKILL mid-response, (3) resume after hook-exit-2 blocked stop, (4) resume across two Node processes, (5) resume when the worktree has a sibling `claude` session history, (6) resume with a new `-p` prompt on stdin. Only after this should the retry semantics be finalized. Also: define fallback behavior when `-c` fails — start a fresh session with a "you were previously working on X, here's the failure context" preamble. The config option should be `retry_mode: continue | fresh | continue_then_fresh`.

### "Retry" is not meaningful inside a single session with a blocking Stop hook
**Issue:** Phase 2 § Retry loop says `max_retries: 3` with `retry_mode: continue`. But if the implementer hits a Stop hook that keeps failing (exit 2, stderr fed back), Claude Code itself already loops — the agent fixes, tries to stop, hook blocks, agent fixes again. The harness only sees the session when it finally exits (either cleanly or because the agent gave up). "Retry 3 times" from the harness's perspective means "run a new `claude -c` after the agent declared defeat 3 separate times" — which is a fundamentally different thing from the inner hook-retry loop. The plan conflates these.
**Impact:** Users will set `max_retries: 3` expecting 3 attempts to fix the code and get 3 outer-loop re-invocations on top of however many inner hook retries already happened. The meaningful budget (total tool calls? total tokens? inner retries?) is invisible.
**Proposed change:** Rename/clarify: `max_outer_retries` (harness re-invokes after the session ends in failure) vs. `max_inner_hook_retries` (Stop hook's own counter — implemented in the hook script, since the hook has no persistent state across calls unless we give it one). Document in the plan that the Stop hook should write a counter to a file in the worktree and give up after N inner retries with exit 0 + a message, letting the session end so the harness can decide what to do. Also define: what does a "retry" prompt look like? The plan says "send the failure context back" but doesn't define what that is — stderr of the failing hook? The last N stream-json events? A manually assembled summary?

### Hook exit code semantics need verification and one-sided coverage
**Issue:** The Hooks Integration section asserts "exit 2 # blocks the stop, feeds stderr back to Claude." This is asserted as fact but not cited, and the plan doesn't cover exit 1 (generic failure — does Claude Code ignore it? treat it as block? treat it as session-fatal?) or non-Stop hooks (PreToolUse blocking semantics differ from Stop). The plan also doesn't specify the hook's stdin contract (Claude Code passes JSON on stdin to hooks in recent versions) or the stdout contract (JSON responses control decision/block behavior).
**Impact:** The entire quality-gate story depends on hook behavior that isn't pinned down. If exit 1 from a hook silently kills the session, the require-passing-checks.sh script as written will cause pipeline deaths instead of retries.
**Proposed change:** Add a "Hook Contract" subsection that verifies (needs verification — see Questions) and documents: exact exit code mapping for each hook type (PreToolUse, PostToolUse, Stop, SubagentStop, UserPromptSubmit), the stdin JSON schema Claude Code passes to the hook, the stdout JSON schema for structured responses, and what happens when a hook itself crashes (killed by signal, writes malformed JSON, takes >30s). The plan should cite the Claude Code hook docs, not assert behavior.

### Worktree creation has unhandled race conditions and failure modes
**Issue:** Worktree Management assumes happy-path create/bootstrap/cleanup but doesn't address: (a) two parallel workflows racing on branch names (`harness/add-auth` collision), (b) bootstrap failure mid-run (half-installed node_modules, containers started by `docker compose up -d` that the cleanup doesn't know about), (c) user has uncommitted changes in the source working tree — does `git worktree add` silently inherit them? (it doesn't — worktrees share the repo but each has its own index/HEAD, except that staged changes in the main worktree don't move, but `.gitignore`'d files like `node_modules` and `.env` do NOT appear in the new worktree and must be recreated by bootstrap), (d) cleanup on a worktree whose branch has unmerged commits (`git worktree remove` refuses without `--force`), (e) cleanup racing with a still-running child that holds file handles.
**Impact:** First failure scenario corrupts state, second leaves orphan containers, third confuses users ("I had my .env set up, why doesn't the agent see it"), fourth makes `auto_cleanup: true` silently fail, fifth corrupts the next run in the same workflow.
**Proposed change:** (1) Branch name collisions: suffix with workflow id (`harness/add-auth-<shortid>`), never reuse. (2) Bootstrap atomicity: treat bootstrap as its own phase in the state machine with its own status; if it fails, mark the workflow `bootstrap_failed` and do NOT auto-cleanup (user needs to inspect). (3) Explicitly document in config guide that bootstrap is responsible for reproducing any gitignored state (env files, secrets, containers) and provide a `.harness/teardown.sh` hook for reverse cleanup (stop containers, etc.), run before `git worktree remove`. (4) Cleanup must `kill` any tracked child pids for that workflow first, then run teardown, then `git worktree remove --force`, then handle the "branch not merged" case (either delete branch or leave it — configurable). (5) Never auto-cleanup a worktree whose branch has unpushed/unmerged commits unless the PR was created successfully.

### SQLite schema has no migration story and missing indexes
**Issue:** Crash Recovery § State Persistence shows three CREATE TABLE statements. There is no migrations system, no `schema_version` table, no discussion of upgrade paths. There are no indexes declared — but `sessions` will be queried by `workflow_id` and `feature_id` on every state transition and every UI refresh, and `features` by `workflow_id` for the feature board. WAL mode is mentioned ("WAL for concurrent reads") but not how it's enabled, nor the gotchas (sidecar files, fsync on commit, recovery after crash).
**Impact:** First schema change breaks every existing deployment with no upgrade path. UI gets slower over time as session logs accumulate. WAL sidecar files left after a crash surprise users.
**Proposed change:** (1) Add a `schema_migrations` table and a simple forward-only migration runner (list of SQL files or in-code migrations keyed by version). Run migrations on startup inside a transaction. (2) Declare indexes: `CREATE INDEX idx_features_workflow ON features(workflow_id)`, `CREATE INDEX idx_sessions_workflow ON sessions(workflow_id, feature_id)`, `CREATE INDEX idx_workflows_status ON workflows(status)`. (3) Enable WAL explicitly (`PRAGMA journal_mode=WAL`) and document: (a) WAL requires the DB directory to be writable, (b) `.db-wal` and `.db-shm` sidecar files exist and should not be committed, (c) use `PRAGMA synchronous=NORMAL` for the write-heavy orchestration path (still safe with WAL). (4) Wrap every state transition in `db.transaction(() => {...})()` and document that reads during writes are fine but that the UI must use a separate connection for read-only queries (or serialize everything through the writer). (5) Add an `events` append-only log table for debugging state machine transitions — this is worth it from day one.

### Prompt template "variable interpolation" is unspecified and is a shell-injection vector if done naively
**Issue:** Configuration § Prompt Templates says "Templates use variable interpolation" and shows `{{feature_spec}}` without naming an engine. The plan also doesn't address: escaping (what if `feature_spec` contains `{{something}}`?), missing variables (crash, warn, leave as literal?), nested references, conditionals/loops (is Mustache enough? do we need Handlebars?). The "shell injection" angle is that prompts are assembled into a temp file and piped to stdin — not directly to a shell — so classic injection is not the risk. But the plan also shows `check_command: "claude --usage --output-format json"` as a string that presumably gets shell-exec'd, and the bootstrap `commands: - "pnpm install"` list is similar. Those ARE shell-exec'd and become injection vectors if any variable gets interpolated into them.
**Impact:** Template engine choice gets made ad hoc in week one and becomes a compatibility constraint forever. If any config field with interpolation is ever passed to a shell, we get command injection.
**Proposed change:** (1) Pick a template engine up front. Recommend a minimal Mustache-style engine (e.g., `mustache` or a 30-line hand-rolled `{{name}}` replacer). Explicitly NO code execution in templates. (2) Define variable resolution rules: undefined variable is a hard error at assembly time (fail the phase, show which variable, which template), never silently empty. (3) Define escaping: no escaping needed for prompt content because it's not going through a shell. (4) The bootstrap `commands` list and `check_command` must be spawned with `spawn(cmd, args, {shell: false})` where possible, and NEVER have any harness-managed variable interpolated into them in v1. If interpolation is needed later, spawn with `shell: false` and pass args as an array. (5) Document the template variable inventory explicitly in the plan (feature_spec, architecture_md, progress_md, git_log_recent, feature_id, workflow_name, ...) so phase authors know what's available.

### Feature dependency ordering is in v1.1 but is load-bearing for v1
**Issue:** Requirements § Should Have (v1.1) lists "Feature dependency ordering (topological sort of `depends_on`)". But `features.json` already has a `depends_on` field in v1 (File Contract § features.json). If the planner emits features with dependencies and the harness runs them in JSON order, a feature can start while its dependency is still `pending`, the prompt won't include the dependency's code because it doesn't exist yet, and the implementer will reinvent or block. The state machine also needs to handle "feature A failed → features that depend on A should auto-block."
**Impact:** v1 either ignores the `depends_on` field (misleading — users will set it and wonder why ordering is wrong) or crashes when it's non-empty. Dependency cascade-blocking is absent.
**Proposed change:** Move topological sort into v1. It's a 20-line function and it's cheaper to build now than to retrofit. At minimum in v1: (a) topo-sort features before implementation phase, (b) on feature failure, mark all transitively dependent features as `blocked` with `blocked_reason: "dependency feat-NNN failed"`, (c) reject cycles at planning-validation time with a clear error. Advanced parallel scheduling of independent features can still be v1.1.

### Heartbeat "output activity in last N minutes" is the wrong signal during long tool calls
**Issue:** Process Management § Heartbeat says "Has stdout emitted anything in the last N minutes? (generous default — maybe 5 minutes)." But stream-json emits nothing while a single `Bash` tool call is running — e.g., a 12-minute integration test suite, a long `pnpm install`, a docker build. 5 minutes will false-positive on any real build. The plan acknowledges this ("a process might legitimately be doing a long build") but then still proposes the timer.
**Impact:** Harness kills legitimate long-running work, or the user sets N so high that the heartbeat is useless.
**Proposed change:** Distinguish two signals: (1) **Liveness** — PID still running AND, if the session is in a tool call, the tool's child process is still running. Use the stream-json tool_use events to track "we're currently inside tool X, started at T." Never alert on stdout silence while a tool is executing. (2) **Stream activity** — only apply a stdout-silence threshold when NOT inside a tool_use. (3) Make the "kill after silence" action user-confirmed by default — heartbeat surfaces a warning in the UI, does not kill. The user clicks "kill and retry" if they judge it stuck. This matches the plan's "user can judge whether the agent is stuck" intent but actually implements it.

### Token usage parsing from stream-json is underspecified
**Issue:** Usage Tracking shows a `{input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens}` shape as the capture target, but doesn't specify which stream-json event carries the usage numbers, whether it's per-turn or cumulative, whether it's updated continuously or only at session end, or how to reconcile when a session is `-c`-resumed (is the usage for the original + resumed, or only the resumed slice?).
**Impact:** Token reporting is either wrong (double-counting resumed sessions) or silently empty (we're parsing the wrong event).
**Proposed change:** (needs verification — see Questions). Specify: which event type(s) carry usage, accumulation semantics, and how resumed sessions are linked in the DB. Proposed DB shape: add `parent_session_id TEXT REFERENCES sessions(id)` so a resumed session can chain; per-session usage is the delta from that session only; per-feature usage is the sum across the session chain.

### Config validation via ajv + YAML has known footguns the plan doesn't address
**Issue:** Tech Stack names `yaml + ajv` but Configuration § doesn't address: (a) YAML number-like strings (the infamous `version: 1.10` parsing as 1.1), (b) YAML boolean surprise (`on`, `off`, `yes`, `no` in YAML 1.1 vs 1.2 — the `yaml` package defaults to 1.2 but still), (c) ajv error messages are cryptic by default ("must match exactly one schema in oneOf"), (d) schema evolution across versions, (e) unknown-key behavior (typos in config keys silently ignored unless `additionalProperties: false` is set everywhere).
**Impact:** First user to typo a config key loses a day wondering why their setting doesn't work. First user with a `version: 1.10` release in config gets 1.1. Every validation error is unreadable.
**Proposed change:** (1) Require `version: "1"` at the top of `.harness.yml` (quoted string) and validate against it — gives us schema evolution from day one. (2) Set `additionalProperties: false` on every object schema. (3) Use `ajv-errors` + `better-ajv-errors` (or a custom formatter) to produce errors like `worktrees.base_dir: expected string, got null (line 12)`. (4) Document the YAML 1.2 assumption and add a lint-time warning for bare `on/off/yes/no` values (they're strings in 1.2 but users expect them to be booleans). (5) Validate file paths at load time (templates exist, schemas exist) — not at phase-run time.

### GitHub integration "octokit + gh fallback" glosses over auth and the actual fallback trigger
**Issue:** Tech Stack says "Octokit + gh CLI fallback" and the dashboard section mentions "If auto-PR fails or is disabled, user clicks to create via `gh` CLI." This conflates two different things: (a) using gh as a fallback when octokit fails, (b) using gh when the user clicks a button. It also doesn't address auth: does the harness read `GITHUB_TOKEN` from env? from `gh auth token`? from `~/.config/gh/hosts.yml`? What about GHES? What about rate limits? What about the PR creation happening from a detached worktree branch that isn't pushed yet (need to `git push` first — who does that)?
**Impact:** Auth story is undefined, will ship broken on first non-laptop user. Branch-push-before-PR is a separate step that's invisible in the plan.
**Proposed change:** (1) Auth resolution order: `GITHUB_TOKEN` env, then shell out to `gh auth token`, then error with clear guidance. Document this explicitly. (2) Separate the two flows: "auto-PR" uses octokit by default (falls back to gh if octokit errors with a well-defined set of auth/network errors); "manual PR button" always uses gh (because the user has gh configured and we know it works). (3) Explicit pre-PR step: `git push -u origin <branch>` must happen before PR creation; surface push failures (auth, protected branch) distinctly from PR-creation failures. (4) Defer GHES support to v2 and say so.

### "Workflow in_progress" recovery doesn't specify what phase to resume in
**Issue:** Crash Recovery § On Harness Restart says "For any workflow that was `in_progress`: check if the worktree exists... attempt `claude -c`... if that fails, start a fresh session for the interrupted phase." But the plan's state model stores `status` on workflows and features, and `current_phase` on features. It doesn't store which session was active, which subagent, whether a hook was in the middle of blocking a stop, or whether the agent was mid-commit. A crash during a commit leaves a dirty worktree the next session will choke on.
**Impact:** Resume lands in an inconsistent state and the fresh session gets confused by uncommitted changes from the crashed session.
**Proposed change:** (1) On every stream-json event of interest (assistant turn start, tool call start/end, hook fire), update `sessions.last_event_at` and `sessions.last_event_type` so restart knows roughly where we were. (2) On restart, before attempting `-c`, inspect the worktree: if `git status` is dirty, either (a) auto-stash with a `harness-crash-stash-<timestamp>` name and surface it in the UI, or (b) commit as WIP with a `[harness-crash-wip]` prefix — configurable. Never run `-c` against a dirty worktree without preserving the work first. (3) For features in `review` state when the crash happened, the resume must know whether the review phase had already written files (check `reviews/feature-N/` for partial files and delete them so the re-run starts clean).

### `--dangerously-skip-permissions` as default is a policy decision the plan makes silently
**Issue:** Phase 2 § Permission mode says `--dangerously-skip-permissions` is the default (configurable per phase). This is a real decision with real blast radius — the agent can `rm -rf`, modify files outside the worktree (worktree isolation doesn't stop `~/.ssh/config` writes), and there's nothing stopping a prompt injection in the planning spec from escalating. The plan doesn't discuss the PreToolUse hook as a mitigation (the table mentions "Dangerous commands blocked | PreToolUse hook" but doesn't follow through).
**Impact:** First prompt-injection incident destroys something outside the worktree and the harness is blamed.
**Proposed change:** (1) Keep `bypassPermissions` as the *default for implementer*, but make the harness install (or require) a PreToolUse hook that denies writes/reads outside the worktree path and denies a curated list of dangerous Bash patterns (`rm -rf /`, `chmod -R 777`, writes to `~/.ssh`, `~/.aws`, `~/.config`, `/etc/`, etc.). Make this hook ship as part of `harness init` scaffolding so every new project gets it. (2) Document the threat model explicitly in the plan: "The planner spec is untrusted input. The harness must not assume the implementer won't try to escape the worktree." (3) Offer a config flag `safety_mode: strict | default | yolo` that toggles the hook set.

## Questions (need user input)

- Needs verification: What is the exact framing of Claude Code's `--output-format stream-json` output? NDJSON (one JSON object per line, `\n`-delimited)? Concatenated objects with no delimiter? Some event types multi-line? The parser design depends on this. Can you point me at the stream-json docs or a captured transcript I can inspect?

- Needs verification: Exact semantics of `claude -c`. Specifically: (a) does it work across separate Node processes (harness restart)? (b) does it work after SIGKILL of the prior `claude` process? (c) does it accept a new prompt on stdin, or does it ignore stdin and just continue from the last state? (d) is "the last session in this directory" stored where — `.claude/` in the worktree, or user-global? (e) does `jig run <profile> -- -c ...` preserve session state if the profile changes between calls? This is the single most load-bearing assumption in the plan.

- Needs verification: jig-specific gotchas. What does `jig run <profile> --` actually do with cwd (does it honor the caller's cwd or does jig cd somewhere)? Does jig fork its own child or exec? Does jig propagate signals to the claude child? Is there a known "spawn jig from Node" pitfall (shell alias vs. binary on PATH, `.tool-versions` resolution, profile-not-found exit codes)?

- Needs verification: Hook exit code semantics for each hook type. The plan asserts `exit 2` on a Stop hook blocks and feeds stderr back. Is that documented? What does exit 1 do? What does exit on signal do (hook killed)? Does Claude Code pass JSON to the hook on stdin, and does structured JSON output from the hook override exit codes? Are hooks invoked per-session or per-matcher-match, and can a Stop hook see the session transcript?

- Needs verification: Does Claude Code emit token usage in stream-json incrementally (per turn) or only in a terminal event? If per-turn, is it cumulative or delta? How does `-c` resume affect the usage numbers emitted — do we see only the new slice, or the re-hydrated totals?

- Should the harness own the PreToolUse safety hook, or is that delegated to the project's own `.claude/hooks`? If the harness owns it, does `harness init` scaffold it and how do we version-upgrade it when users have customized it?

- What's the intended relationship between the harness's retry loop and any retry logic inside the Stop hook itself? Is the hook expected to be stateless (retry forever until the agent gives up) or stateful (maintain a counter file and exit 0 after N)? This determines whether "max_retries: 3" in config means inner or outer.

- For parallel workflows: is the intent that multiple workflows can target the same base branch concurrently, or is there a lock? If concurrent, how do we handle the case where workflow A's PR gets merged while workflow B is still running off a now-stale base?

- The plan mentions `lazyworktree` as a possible cleanup tool — is that a real tool I should integrate against, or a placeholder? Do you have a preferred cleanup contract (stdin/stdout? argv? exit codes?) for the `cleanup_tool: custom` case?

- Session logs: `retain_stream_json: true` captures everything. What's the retention policy — forever? Rotate after N days/MB? Who deletes? This matters for the schema migration and disk-space story.

- Is the v1 dashboard single-user (localhost, no auth) or multi-user? The plan says "runs locally" but also mentions "browser push" notifications. If it's ever remote, we need an auth story in v1, not v2.
