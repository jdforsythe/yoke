# QA Critique — plan-draft2.md

## Accept (already correct)
- **State persisted on every transition before it takes effect.** This is the right invariant for crash recovery. If honored rigorously (write-ahead, fsync-before-ack), the SQLite store becomes the authoritative timeline. Don't let this slip to "persisted sometime soon."
- **File-based agent communication.** Removes a huge class of concurrency bugs and makes the system replayable from artifacts alone. This is also the single biggest lever for test determinism — see Challenge below on using it.
- **Heartbeat + stream-based activity detection instead of wall-clock kill timers.** Correct instinct. A 20-minute test run is not a stuck agent.
- **Quality gates in hooks, not the harness.** Avoids double-running expensive suites and keeps the harness from needing project-specific knowledge.
- **Per-feature `review_criteria` written by the planner.** This is the correct anti-rubber-stamp mechanism; combined with schema validation of the review JSON it gives the harness something concrete to enforce.
- **WAL-mode SQLite with synchronous API (better-sqlite3).** Appropriate for single-writer state, and survives concurrent readers from the dashboard.
- **Structured stream-json capture per session.** Gives you a replayable artifact, which is the foundation for everything in the "Challenge" section below.

## Challenge (propose revision)

### No mocking strategy for jig / Claude Code — the plan is untestable as written
**Issue:** The entire v1 checklist assumes a real Claude Code binary driving real file edits. There is no abstraction for jig, no "fake agent" harness, no replay mode from captured stream-json. Every E2E test therefore costs real tokens, takes minutes-to-hours, and is non-deterministic. CI cannot run it. Developers cannot run it on a PR.
**Impact:** You cannot prove the state machine, retry loop, hook integration, or crash recovery work without burning a Max subscription on every test run. Regressions will ship. Reviewers will stop running the suite.
**Proposed change:** Introduce a `ProcessManager` interface with two implementations: `JigProcessManager` (production) and `ScriptedProcessManager` (test). The scripted manager reads a pre-recorded stream-json JSONL file and replays it with configurable timing, injected exit codes, and injected stderr. Add a `yoke record` mode that captures a real session's stream-json + artifact file deltas into a fixture. The v1 test suite then runs entirely on fixtures: happy path, hook-failure-then-retry, rate-limit-midstream, SIGKILL-midstream, malformed stream-json, etc. Put "ScriptedProcessManager + at least one fixture per phase" in Must-Have v1.

### Crash recovery has no test strategy and no crash-injection surface
**Issue:** The plan says "on restart, load state and attempt `-c` resume" but never says how this is verified. You cannot wait for real crashes. There is no fault-injection hook, no "die between persist and ack" test, no torn-write test, no "worktree exists but SQLite thinks it was cleaned" test.
**Impact:** Crash recovery is the feature most likely to be broken and least likely to be noticed until a user loses a day of work. "Persist before transition" is easy to write and easy to get wrong (async fire-and-forget, missing fsync, transition enum that was added after the persistence layer).
**Proposed change:** Add an explicit `FaultInjector` seam that can be configured to `panic`, `exit`, `kill-9-child`, or `corrupt-next-write` at named checkpoints (`before_persist`, `after_persist_before_spawn`, `after_spawn_before_ack`, `during_artifact_read`, etc.). Write tests that drive the pipeline, trigger a fault at each checkpoint, then invoke the recovery path and assert the workflow converges to the same terminal state as an uninterrupted run. Require fsync on the SQLite transaction that records each transition. Add a startup integrity check that reconciles SQLite state against worktree state and surfaces discrepancies.

### State machine is described in prose, not defined
**Issue:** Statuses `pending | in_progress | review | complete | blocked` are listed once. There is no transition table, no guard conditions, no list of illegal transitions, no definition of what happens on every `(state, event)` pair. Specifically: what happens when a feature is `in_progress`, the implementer crashes, and on restart the worktree has commits that weren't recorded? What happens when a review comes back partially `fail` and the retry limit is hit mid-phase? What does `review` → `review` look like on re-review? Is there a `retrying` state distinct from `in_progress`?
**Impact:** Every undefined `(state, event)` becomes a production bug. Deadlock is likely: a feature can sit in `review` forever if the reviewer fails to produce a valid JSON and no path forward is defined.
**Proposed change:** Replace the prose with an explicit transition table (`from_state × event → to_state, side_effects, guard`) committed as a YAML or TS const, with a unit test that asserts every state has a defined transition for every legal event and every other event is a well-defined error. Add states for `awaiting_retry`, `awaiting_user`, `abandoned`, and `review_invalid` (reviewer produced unparseable output). Every terminal-ish state must have an escape hatch (manual unstick) from the dashboard.

### "Blocked" is a black hole with no recovery path
**Issue:** `max_retries` exhausted → feature marked `blocked`. The plan stops there. There is no "retry later," no exponential backoff for transient failures, no distinction between transient (rate limit, network, flaky test) and permanent (impossible spec, contradiction with architecture) failures. No defined user action to un-block. No way to bump the retry counter from the UI.
**Impact:** A single flaky test will terminally block a feature. A user who nudges the spec cannot resume. Long-running workflows will slowly bleed out into a pile of `blocked` features.
**Proposed change:** Introduce a failure classifier that tags each failure as `transient`, `permanent`, `policy`, or `unknown`, with the default being `unknown` → `awaiting_user`. Transient failures use exponential backoff and a retry budget per time window, not a monotonic counter. `blocked` becomes a user-reachable state with a "retry with notes" action that spawns a fresh implementer with injected context. Every blocked feature must have a machine-readable `blocked_reason` and a human-readable explanation.

### "No arbitrary timeouts" conflates two failure modes
**Issue:** The plan correctly rejects "kill after N minutes" but then has nothing to say about a *silent* wedge: the agent made a tool call, got no response, is waiting on an MCP server that will never answer. stream-json emits nothing. The heartbeat's "5 minute default" is the only signal, and that's both too long to matter and too short for legitimate builds. "The user will watch the dashboard" is not an operable answer for an overnight autonomous workflow.
**Impact:** Workflows silently wedge overnight. User wakes up to zero progress, no notification, no forensic trail.
**Proposed change:** Distinguish (a) wall-clock timeouts (rejected, correctly) from (b) *activity* timeouts (needed). Track per-tool-call latency from stream-json events (`tool_use_start` without a corresponding `tool_result` within N seconds is a wedge signal). Emit a "stalled" notification, not a kill, at the configurable threshold. Separately, track cumulative phase wall time and emit a "long-running" notification at a configurable threshold — again, notification, not kill. User-initiated cancel remains the only kill path, but they must be *told* when to consider it.

### Hook failure modes are under-specified
**Issue:** The plan covers `exit 2 blocks stop`, and nothing else. What if the hook script is missing (`ENOENT`)? What if it exits 1 instead of 0 or 2? What if it hangs forever because a test-runner child deadlocked? What if it prints gigabytes to stderr? What if it succeeds but leaves the worktree in a dirty state? What if the hook is intentionally sabotaged by the agent itself (agent has Edit + Bash, can rewrite the hook file)?
**Impact:** Silent-pass (hook broken → agent "succeeds" with failing tests), silent-hang (hook wedged → phase wedged), or prompt-injected bypass.
**Proposed change:** Harness validates hook script existence and executability at workflow start and fails fast. Harness also independently re-verifies the hook's claimed invariants post-phase (at minimum: `git diff` is non-empty, at most a fast `pnpm run typecheck` as a tripwire). Hook stdout/stderr captured with a hard byte cap. Hook wall-clock is the *one* place arbitrary timeout is appropriate — hook is a guard, not agent work, and a hung hook should be killed and reported. Protect hook scripts: store a checksum of the hook file at workflow start and verify it unchanged at Stop; diverging checksum is a hard fail requiring user review.

### Stop-hook-as-gate is not actually tamper-proof
**Issue:** The plan's central insight — "by the time the session ends, checks have passed" — holds only if the agent cannot alter the checks. With `--dangerously-skip-permissions` the agent can `rm` the failing test, `git add -A`, comment out the assertion, add `test.skip`, edit `.claude/hooks/require-passing-checks.sh` to `exit 0`, or edit `package.json` scripts so `pnpm run test` is a no-op. All of these will make the hook "pass." The reviewer phase might catch some of it but the plan also says the reviewer does *not* re-run checks.
**Impact:** The quality gate is a polite suggestion. A misaligned or confused agent silently ships broken code.
**Proposed change:** (1) Pin the hook script and `package.json` scripts via checksum at workflow start; Stop hook refuses if checksums changed and no explicit approval event was recorded. (2) Stop hook must output a signed manifest of `{test_count, test_pass_count, file_checksums_of_test_files}` that the harness parses and sanity-checks against the prior session. (3) Reviewer explicitly verifies that test files changed *alongside* production files (an anti-skip check) and that no test was deleted without justification. (4) Add a "skeptical re-run" mode where the harness does run the test suite itself for a subset of features (e.g., final feature, or spot-checked at random). The "harness doesn't redundantly re-run" principle is a nice optimization; correctness wins.

### Observability is thin — no correlation IDs, no trace, no structured log schema
**Issue:** "Session logs are stored" and "dashboard shows live output" are not observability. There is no mention of: a workflow-scoped correlation ID that propagates to every log line, a feature-scoped child ID, a phase-scoped run ID, a structured log schema, a trace of `(workflow_id, feature_id, phase, session_id, attempt)` on every record, or a query interface. Debugging a stuck workflow a week after the fact will require grepping raw JSONL.
**Impact:** Production triage is infeasible. "Why did feat-017 take 9 retries?" becomes a half-day excavation.
**Proposed change:** Define a structured log schema with mandatory fields `{ts, workflow_id, feature_id, phase, session_id, attempt, event_type, level, message, ...event_specific}`. Every child process gets these as env vars and every harness log line is emitted through a logger that enforces them. Add a `GET /workflows/:id/timeline` endpoint that returns the merged event stream from SQLite + per-session JSONL. Put this in v1, not v1.1 — session log viewer is listed as v1.1 and that's wrong: you cannot ship v1 without a way to debug v1.

### Failure enumeration is not in the plan at all
**Issue:** The plan lists what the harness *does*, not what breaks. Missing from any section: jig not installed, jig profile missing, worktree creation fails (disk full, base branch missing, uncommitted parent-repo changes), SQLite locked or corrupt, git worktree prune killed our branch, bootstrap command fails (network, private registry, `docker compose up` conflict on port), rate limit hit mid-stream, OAuth token expired mid-stream, process killed externally (OOM killer, user `pkill`, laptop sleep), GitHub API down, GitHub CLI unauthenticated, prompt template file missing, prompt template references undefined variable, features.json schema changed between plan and implement, two workflows compete for the same branch name, clock skew between events.
**Impact:** Every unenumerated failure is a production incident with ad-hoc handling.
**Proposed change:** Add a "Failure Modes" section with a table: `{failure, detection, user-visible outcome, recovery action, test fixture}`. Require every failure mode listed there to have a fixture in the test suite. Block v1 release on all of them having defined outcomes (a defined outcome of "surface to user and wait" is fine; undefined is not).

### Concurrency and shared-resource contention not addressed
**Issue:** "Named workflows with parallel support" is a v1 Must-Have. The plan does not discuss: the SQLite writer being a single mutex point under parallel workflows, jig profiles potentially sharing MCP server state, shared `.claude/` directory being mutated by parallel sessions, the hook script running the same port-binding `docker compose up` twice, git's refusal to worktree the same branch twice, bootstrap side effects (`pnpm install` in a shared global cache), token rate limits being a *global* resource silently shared across workflows.
**Impact:** Parallel workflows race, corrupt each other's state, or deadlock on global resources.
**Proposed change:** Define the set of global resources (`.claude/`, the MCP server processes, the SQLite writer, the Claude account rate budget, the git index if worktrees share it, bootstrap ports) and pick one of: hard-serialize via a named lock, document as "you must not do this," or isolate per-workflow. Add a "concurrency contract" section. Add a stress test that runs 5 parallel scripted workflows and asserts all reach terminal states without corruption. Decide whether v1 ships parallel at all — it might be safer to ship v1 single-workflow and move parallel to v1.1.

### Disk and retention have no policy
**Issue:** Worktrees, session JSONL, SQLite, stream-json captures, review JSON files all grow unboundedly. `retain_stream_json: true` is a flag but no counterpart retention setting. A multi-week workflow on a laptop will fill the disk.
**Impact:** Disk-full crashes the harness, corrupts SQLite mid-write, kills the agent mid-tool-call. Classic cascading failure.
**Proposed change:** Add `retention:` config: max age, max total size, per-workflow cap. Add a disk-space preflight before each phase that refuses to start if free space is below a threshold. Add a nightly compaction job (rotate JSONL, vacuum SQLite). Instrument disk usage in the dashboard. Put a "disk full mid-write" fixture in the test suite.

### Security threat model is entirely missing
**Issue:** `--dangerously-skip-permissions` is the default. Prompt-injection via the user-provided spec file is unaddressed. The spec could instruct the planner to write an `init.sh` that exfiltrates `~/.ssh`, or instruct the implementer to `curl | sh`. Hooks run arbitrary shell. Worktrees run arbitrary shell. The plan does not identify who the adversary is, what the trust boundaries are, or what the harness is supposed to prevent.
**Impact:** A malicious or compromised spec (pasted from a GitHub issue, forwarded by a colleague) can own the developer's machine. An autonomous overnight workflow multiplies the blast radius.
**Proposed change:** Add a "Threat Model" section naming: (a) spec content is untrusted input; (b) the agent may attempt actions outside its scope; (c) the developer machine is the trust boundary. Define a `PreToolUse` deny-list for dangerous ops (`curl|sh`, writes outside worktree, ssh key access, env dumps, network egress to non-allowlisted hosts). Run agents under a subuser or sandbox (firejail, sandbox-exec on macOS) when feasible. Surface and require approval for any `init.sh` on first run. Require explicit opt-in (not default) for `--dangerously-skip-permissions`, or at least gate it on a per-phase, per-workflow confirmation the first time a new spec is run.

### The "no message passing" principle will leak in practice
**Issue:** Reviewer needs to know which files the implementer *intended* to change vs. touched incidentally. Reviewer needs to know which acceptance criteria the implementer consciously deferred. Re-implementer needs to know which specific review points matter most. `progress.md` is free-form and not a contract. `git log` is not enough. In practice, QA reviewers will need cross-phase context that isn't in the artifacts, and the retry path especially — where `-c` reuses the prior context window — violates the stated principle anyway.
**Impact:** Either (a) reviewers hallucinate the missing context, or (b) users hack it in via the "inject context" feature, which becomes the unofficial message bus with none of the guarantees.
**Proposed change:** Own the leak. Add a structured `handoff.json` per feature with `{intended_files, deferred_criteria, known_risks, retry_history[], reviewer_notes_seen[]}`. Keep it append-only. Document that this *is* the cross-phase channel and that context-injection goes through it. This gets you testable handoffs without reopening message passing.

### v1 acceptance checklist is a feature list, not a release gate
**Issue:** The Must-Have list is what v1 *contains*, not what must be *true* to ship. There is no "v1 is done when..." section. No list of scenarios that must pass. No production readiness review.
**Impact:** v1 will ship when someone is tired, not when it's ready. No one will agree on "is it ready."
**Proposed change:** Add a "v1 Acceptance" section with a concrete scenario list the QA phase can mechanically run: (1) scripted happy-path workflow completes in <5s; (2) SIGTERM mid-implement → restart → resumes to same terminal state; (3) hook-failure-then-retry converges; (4) hook-exit-nonzero-non-2 fails cleanly with clear error; (5) 5 parallel scripted workflows all complete without corruption; (6) disk-full during session produces a clean, resumable failure; (7) malformed features.json is caught by schema validation with a helpful error; (8) worktree creation failure surfaces clearly; (9) crash during SQLite write is recovered via WAL; (10) `blocked` feature can be manually retried from UI. Release gate = all green.

### Retry semantics: `-c continue` will silently poison retries
**Issue:** `retry_mode: continue` uses `-c` to resume the session with accumulated context. That context includes the failure the agent just produced. The 4th attempt sees the prior 3 failed attempts in its context window, which is great for learning from mistakes and *terrible* for token usage, context-window exhaustion, and anchoring bias (agent keeps trying the same broken approach). There's no branch-and-reset option.
**Impact:** Retries degrade quality over attempts. Token cost explodes. Context eventually overflows with a cryptic error.
**Proposed change:** Retry strategies should be explicit: `continue` (default, cheap), `fresh_with_failure_summary` (harness writes a summary to `handoff.json`, spawns a new session), `fresh_with_diff` (new session sees the diff but not the prior agent's reasoning). Make the strategy configurable per phase and escalate automatically: attempt 1 = `continue`, attempt 2 = `fresh_with_failure_summary`, attempt 3 = `awaiting_user`. Test each path with scripted fixtures.

### Rate-limit detection and reaction is hand-waved
**Issue:** "If the user hits rate limits, Claude Code's own behavior handles it (the process pauses or fails)." Which is it — pauses or fails? They are very different. The harness's heartbeat at 5-minute output gap will false-positive a "paused" process as "stalled" and may kill-and-retry into the same rate limit. There is no backoff. No "rate-limited" state. No dashboard indicator.
**Impact:** Harness fights with the rate limiter, burns budget on doomed retries, or silently stalls.
**Proposed change:** Parse stream-json for rate-limit events explicitly. Enter a first-class `rate_limited` state with a known wake-up time (from the error if available, else exponential probe). Suppress heartbeat "stalled" alerts during rate_limited. Show the waiting state in the UI with estimated resume time. Add a fixture that replays a captured rate-limit stream-json event and asserts the harness enters and leaves `rate_limited` correctly.

### Notifications being "fire and forget" hides failures that matter
**Issue:** "If [notifications] fail, log it and move on" is fine for non-critical pings. But the notification about a *blocked* workflow or a *crashed* harness is the user's only signal something is wrong. Fire-and-forget means the one notification that had to land didn't, and the user discovers the problem 18 hours later.
**Impact:** Overnight failures go unnoticed until morning.
**Proposed change:** Classify notifications by severity. `info` is fire-and-forget. `requires_attention` is retried with backoff across multiple channels and persisted as a "pending user attention" item in the SQLite store and dashboard until acknowledged. The dashboard itself is a pull-based fallback — the "requires attention" banner is authoritative, not the push notification.

## Questions (need user input)
- What is the intended execution environment for v1 — a developer laptop that goes to sleep, a dedicated machine, or both? Laptop-sleep handling (process paused, clock jumps, network reappears) is a significant design input and the answer changes crash-recovery priorities.
- Is parallel-workflow support actually needed in v1, or can it move to v1.1? Shipping parallel requires the concurrency contract, stress test, and shared-resource isolation work above. Sequential v1 halves the failure surface.
- What's the trust model for user-provided specs? Always written by the operator, or possibly pasted from external sources (GitHub issues, Slack)? This determines how aggressive the prompt-injection mitigations need to be.
- When `max_retries` is exhausted and a feature is `blocked`, should the *workflow* continue with other features or halt? The plan implies continue; confirm, because "continue" means a user can wake up to a "completed" workflow that is silently 40% blocked.
- Is it acceptable to re-run test/lint/build once in the harness for spot-checks (the "skeptical re-run" above), or is the "hooks are the single source of truth" principle load-bearing enough that we'd rather harden hook-tamper detection instead?
- Should the `handoff.json` cross-phase channel be introduced in v1, or are we comfortable with `progress.md` + context injection as the MVP and accepting the known leak?
- What does "worktree exists and has uncommitted changes" mean for resume semantics? Does the harness preserve, stash, or discard them? Different answers imply very different user experiences after a crash.
- How should the harness behave when the user's Max subscription's weekly window approaches exhaustion mid-feature? Pause the workflow (requires a pause state and scheduled resume), fail it, or keep going until the hard limit? The "usage gate" is opt-in but the plan doesn't say what happens when the limit is hit with the gate disabled.
- What's the retention policy target — weeks? months? forever? This is a pure product decision that unblocks the retention design.
- Is there any appetite for running agents in a sandbox (macOS sandbox-exec, firejail, or Docker) in v1, or is that strictly a v2 concern? The threat-model mitigations above are much stronger if sandboxing is on the table.
