# Architect Critique — plan-draft2.md

## Accept (already correct)

- **File-artifact communication as the plan/implement boundary (§67-69, Phase 1→2).** Planner writes `features.json` + `architecture.md`, implementer reads them. This is a clean pipe — no shared context window, no backchannel. The boundary is genuinely enforced by the file format, not by convention.
- **Hooks own the quality gates, harness owns the state machine (§71, §289-301).** The "What Hooks Handle vs What the Harness Handles" table is the right split. Keeping tests/lint/build inside the implementer's Stop hook means the harness never has to reimplement a polyglot build runner, and the reviewer never re-runs them. This is the single best structural decision in the draft.
- **Every state transition persisted before it takes effect (§426).** Write-ahead persistence to SQLite is the correct discipline for a long-running supervisor that may be killed mid-operation. Combined with `workflows.status` + `sessions.status`, recovery has a sound basis.
- **Streaming output + heartbeat instead of timeouts (§394-400, §735).** This is the right refusal. Arbitrary kill timers on a tool-using agent are actively harmful; the "user-as-watchdog via streaming" model is architecturally honest about where the judgment lives.
- **Process manager is isolated from pipeline engine (§22-38).** Spawn/stream/heartbeat is a distinct module from the state machine. Good: it means you can unit-test the state machine with a fake process manager and vice versa.
- **Artifact validators as a distinct concern (§98, build step 7).** Schema validation of `features.json` and `reviews/feature-N/*.json` is cleanly scoped at phase boundaries rather than sprinkled through the pipeline engine.

## Challenge (propose revision)

### The Pipeline Engine and the State Store are not actually separate
**Issue:** The architecture diagram (§22-38) shows Pipeline Engine and State Store (SQLite) as distinct boxes, but §426 ("Every state transition is persisted to SQLite before the transition takes effect") and the §429-465 schema describe a pipeline engine whose transitions ARE the SQL writes. There is no in-memory state machine with a checkpointing sidecar — the SQL row *is* the state. That's fine, but then the diagram's separation is a lie and the "State Store" is not a store, it's the engine's backing tape.
**Impact:** Future contributors will try to refactor one without the other, or build a second in-memory cache of workflow state, and the two will drift. The draft never declares whether the SQLite row is authoritative or whether an in-memory object is.
**Proposed change:** Declare explicitly: "The SQLite row is the single source of truth for workflow/feature/session state. The Pipeline Engine holds no durable state in memory — it reads, transitions, writes, and re-reads. Any in-memory object is a short-lived projection." Then collapse the diagram: State Store is not a peer box, it's the persistence layer underneath Pipeline Engine. This also clarifies concurrency: with better-sqlite3 + WAL, readers (dashboard WS) see committed state, which is the behavior you want.

### The Artifact Store box does not exist as a real module
**Issue:** The diagram (§33-37) lists an "Artifact Store (logs, tokens, sessions)". But in the body of the plan, "artifacts" means two completely unrelated things: (a) agent-produced files in the worktree (`features.json`, `progress.md`, `reviews/`), and (b) harness-captured session logs + token usage in `.harness/logs` and the sessions table. These have nothing in common — different writers, different lifetimes, different readers.
**Impact:** Conflating them hides the real dependency graph. File artifacts belong to the worktree (and get cleaned up with it); session logs belong to the harness and must survive worktree cleanup. A reader coming to "Artifact Store" expects one thing and will get confused.
**Proposed change:** Split the concept into "Worktree Artifacts" (read by artifact validators, produced by agents, lives in worktree, deleted on cleanup) and "Session Log Store" (written by process manager, read by dashboard log viewer, lives in `.harness/`, survives worktree cleanup). Remove the "Artifact Store" box from the diagram; replace with "Session Log Store". The artifact validators module reads the worktree directly and doesn't need a store at all.

### The file-artifact principle breaks at the review phase
**Issue:** §676-694 says the reviewer is "an orchestrator session that spawns specialized subagent reviewers" and explicitly notes "Subagent results return summaries to the orchestrator's context (keeps it efficient)." That is exactly the shared-context channel that §69 forbids. Sub-agent verdicts flow through the orchestrator's context window on their way to disk, which means the orchestrator can editorialize, summarize away detail, or (worse) collapse conflicting verdicts into a consensus.
**Impact:** (1) The core principle is no longer "agents communicate through files" — it's "phases communicate through files, within-phase communication goes through context." That's a legitimate model but it's not what §67-69 claims. (2) The rubber-stamping defense in §720-724 is weakened: a security subagent that flags an issue can have it softened by the orchestrator before it lands in `reviews/feature-N/security.json`. (3) Aggregation logic lives inside an LLM, which is the worst place for deterministic pass/fail arithmetic.
**Proposed change:** Either (a) own the contradiction: each subagent writes its own JSON file directly (via its Write tool inside the subagent's scope), the orchestrator never sees the content, and the harness — not the orchestrator — aggregates the files on the filesystem to compute the overall verdict. Or (b) keep the orchestrator but move it from "reviewer" to "review-runner": a thin shell that just spawns subagents in sequence and exits, with no context window at all. My recommendation is (a): the aggregation logic is deterministic (`any(fail) → fail`) and should be code in the pipeline engine, not a prompt.

### The implementer can write to features.json — the rules are a prayer
**Issue:** §190 says "Agents may only change `status`, `blocked_reason`, and `implemented_in_commit`. They must never add, remove, or rewrite feature descriptions..." But the implementer has an `Edit` tool on the whole worktree. There is no structural enforcement of this rule; it's a prompt norm.
**Impact:** An implementer session that goes sideways can silently rewrite acceptance criteria to match what it actually built, and the reviewer — reading the same file — will pass it. This is the same rubber-stamp failure mode §720 is trying to prevent, but at the feature-spec level.
**Proposed change:** Two options. (1) Make `features.json` append-only from the implementer's perspective: store the feature spec in SQLite (you already have `features.feature_data`), not in the worktree, and give the implementer a status-only update path (e.g., a small script in the worktree that calls a harness HTTP endpoint, or a file like `status-updates.jsonl` the implementer appends to). The harness reads that and updates its own `features.json` projection. (2) Post-phase diff check: harness snapshots `features.json` before the implementer session and rejects the session if anything but the three allowed fields changed, forcing a retry. Option (1) is cleaner; option (2) is a v1 shortcut. Either is needed; the current plan has neither.

### Prompt Assembler has a hidden dependency on git and the worktree
**Issue:** The §556-589 prompt template interpolates `{{git_log_recent}}`, `{{architecture_md}}`, `{{progress_md}}`, `{{feature_spec}}`. Three of those come from the worktree filesystem and one comes from `git log`. So the Prompt Assembler actually depends on Worktree Manager AND on git — but in the diagram (§22-38) Prompt Assembler sits next to them as a peer and the dependency isn't drawn.
**Impact:** Hidden coupling. If you ever want to run a dry-run prompt assembly (e.g., for the "preview prompt" dashboard feature that will certainly be requested), you'll find Prompt Assembler can't run without a live worktree. Also affects testability.
**Proposed change:** Define an explicit `PromptContext` interface that the Pipeline Engine builds by pulling from Worktree Manager + State Store + git helper, and passes into Prompt Assembler. Prompt Assembler itself becomes pure: `(template, context) → string`. This matches the architecture diagram's independence claim and makes the real dependency — Pipeline Engine composing all data sources — explicit.

### The phase model is hard-coded as plan → implement → review despite the "configurable pipeline" claim
**Issue:** The config schema (§478-516) declares `pipeline:` as an ordered list and `phases:` as a map, suggesting flexibility. But the state model bakes in three things: (1) `features.json` is produced by "plan" specifically (§98 harness validation); (2) `review.on_fail: implement` is a hard-coded string reference (§515); (3) the SQLite `features.current_phase` column has no schema constraint but the dashboard (§596-600) hard-codes the five statuses `pending / in_progress / review / complete / blocked`. The iterative-planning item in v1.1 (§771) will require the state machine to loop back to `plan`, which the current model has no affordance for.
**Impact:** v1.1's iterative planning is load-bearing for serious use (you cannot plan 40 features in one shot and expect quality), and the current state model will need surgery to support it — which means v1 users will hit a migration. Also: users who want a different shape (e.g., plan → design → implement → test → review, or skip-review mode) will have to fork the state machine code.
**Proposed change:** Decouple the state machine from phase names. The state machine knows three kinds of transitions: (1) phase-complete (artifact produced, validated), (2) phase-failed (retries exhausted), (3) phase-branch (output artifact says "go back to phase X"). Phase names are just labels on nodes in a user-defined graph, not enum values. `features.current_phase` becomes free-text. For iterative planning, the planner writes `features.json` with a flag `needs_more_planning: true` and the pipeline re-enters the plan phase with the current batch as context. This costs very little in v1 (you have to write the state machine anyway) and saves the v1→v1.1 migration.

### Feature dependency ordering in v1.1 is load-bearing for v1
**Issue:** §768 puts "Feature dependency ordering (topological sort of `depends_on`)" in v1.1. But §167 shows `depends_on` already in the v1 feature schema, and §100 ("Implementation (per feature)") implies features run sequentially. Without topological ordering, sequential execution will pick features in JSON-array order, which may violate dependencies, and the implementer will discover it at build-time and either fail or silently do the wrong thing.
**Impact:** Planner outputs are meaningless in v1. Either the planner has to be told "don't emit `depends_on`" (and then the field shouldn't exist in v1) or the engine has to respect it (and then it's v1, not v1.1).
**Proposed change:** Move topological ordering into v1. It's a 20-line implementation on top of the schema you already have. Alternatively, remove `depends_on` from the v1 schema entirely and have the planner emit a flat ordered list. The current middle ground is incoherent.

### Retry via `-c` is a context-recovery mechanism, not a crash-recovery mechanism
**Issue:** §123 ("Retry loop") and §420 ("On Harness Restart → attempt to resume with `claude -c`") both lean on `-c`, but they're using it for two very different things. In §123, `-c` resumes the *same process's* session after a hook failure, which is fine. In §420, `-c` resumes after a *harness* crash — which means Claude Code's own session storage has to survive (a) the original child process exiting abnormally and (b) the harness Node process being killed. That's a Claude-Code-internal guarantee the harness doesn't control, and the fallback ("start a fresh session for the interrupted phase") means the plan needs *two* recovery paths anyway.
**Impact:** Crash recovery quietly depends on an undocumented external contract. When it breaks (and it will, at least once, at a Claude Code version bump), the symptom will be "harness restarted and Claude has no idea what we were doing." The retry-vs-resume distinction isn't drawn anywhere in the plan.
**Proposed change:** Split the concepts cleanly. Call intra-phase retry "continuation" (the `-c` path, used when the same child process is still alive or just died), and call post-crash recovery "resumption" (always a fresh session with the harness reconstructing context from artifacts + progress.md + git log + session log). Treat `-c` as a soft optimization only, never load-bearing. The plan's crash recovery story should not assume `-c` works at all.

### Session log relationship to artifacts is unclear in the schema
**Issue:** The sessions table has `session_log_path` (§462), but there's no table linking sessions to the artifacts they produced or the feature they worked on (beyond `feature_id`, which is nullable). For a plan session that produces `features.json`, how do you know which session produced which feature row? The plan's `features` table stores the data but not the provenance.
**Impact:** Observability ("who wrote this?"), debugging ("which session corrupted features.json?"), and the v1.1 log viewer all need session→artifact joins. Adding them later means backfilling a provenance column.
**Proposed change:** Add `features.created_by_session_id` and `features.last_updated_by_session_id` columns in v1. Optionally add a small `artifact_writes` table keyed by `(session_id, artifact_path, written_at)` if you want full provenance. The cost is minimal now; the retrofit is painful.

### `token_usage TEXT` on sessions is fine but prevents aggregation queries
**Issue:** §461 stores token usage as a JSON blob. §647-652 promises reports "Tokens per feature, per phase, per agent profile, trends over time." You cannot do that efficiently with JSON — every aggregation is a full table scan + JSON parse.
**Impact:** The v1.1 reporting feature becomes either slow or requires a schema migration. Given that you *know* the shape of the token usage object (§638-644), there's no reason to blob it.
**Proposed change:** Promote the four known fields to columns: `input_tokens`, `output_tokens`, `cache_creation_input_tokens`, `cache_read_input_tokens`. Keep a `raw_usage TEXT` column for any extra fields from stream-json that you want to preserve opaquely. This is cheap in v1 and lets v1.1 reporting be pure SQL.

### Design Principle #5 ("Token usage is the user's problem") contradicts the optional usage gate
**Issue:** §76 declares token management a non-goal. §657-672 adds an opt-in per-phase usage gate that pauses the workflow above a threshold. That IS token management. Principles and features disagree.
**Impact:** The principle gives you license to say "no" to future feature requests in this area, but the feature you shipped says "yes." Contributors won't know which to believe.
**Proposed change:** Either drop the usage gate from v1.1 (it's an edge feature anyway and "rate limits pause naturally" is already in §77), or rewrite the principle to say "token usage is observed, not managed — except for a single opt-in pre-flight check." The latter is fine; the former is cleaner.

### The "specialized for Claude Code" stance leaks into the pipeline engine, not just the execution layer
**Issue:** §7 and §75 say Claude Code is the execution layer and the harness wraps it. But Claude-Code-isms are scattered throughout the core: `-c` in the retry loop (§123, §420), `--output-format stream-json` parsing in the process manager (§352, §374), hook-based Stop enforcement as a design assumption (§71), subagent spawning as the review model (§688). These are not in an adapter — they're in the engine.
**Impact:** The "we can add adapters later" escape hatch (§7) is aspirational. In practice, introducing a non-Claude-Code backend would require the state machine to change shape (no Stop hook → where do quality gates live?), the review phase to change shape (no subagents → how does review fan out?), and the retry loop to change shape (no `-c` → what's the continuation model?). The adapter boundary is not drawn anywhere.
**Proposed change:** If v1 is genuinely Claude-Code-only, say so unambiguously: "The harness is not adapter-ready. A non-Claude-Code backend is a rewrite, not a plugin." Remove the "If other agents are supported later, they'll need adapters" line from §7. This is honest and frees the v1 code from pretending to be neutral. If you want adapter-readiness, identify the three specific surfaces above and define protocol interfaces for them now. Half-measures here will be paid for.

### Hook contract has a gap: what happens when the Stop hook's checks aren't available?
**Issue:** §290 says "Tests must pass before implementation is 'done' → Stop hook on implementer." The sample hook (§254-285) is hard-coded to `pnpm run typecheck/lint/test/build`. But the harness is supposed to be reusable across projects, and there's no contract for what happens when a project has no lint step, or uses a different test runner, or is pre-check-in with no tests yet.
**Impact:** The hook is project-specific, which is fine, but the harness has no way to know whether the hook actually enforced anything. A project with a no-op Stop hook gets zero quality gates and the harness happily marks the feature done. The reviewer is the only backstop.
**Proposed change:** Define a minimal hook contract: the Stop hook, on success, must write a small `check-results.json` into the worktree (e.g., `.harness/last-check.json`) listing which gates were run and their outcomes. The harness validates this file before accepting phase completion. If absent, the phase is rejected with "implementer did not declare quality gate results." This keeps the hook project-specific while letting the harness verify that *some* enforcement happened, and it gives the dashboard something real to display.

## Questions (need user input)

- **Is a single planner session sufficient for a 40-feature spec, or do you expect to require iterative planning for real-world use?** If the latter, iterative planning (currently v1.1, §771) is not "should have," it's "must have," and the state machine design needs to accommodate it from day one. I have assumed above that this is load-bearing and challenged the phase model accordingly — confirm or push back.

- **Is `features.json` the canonical feature store or a projection?** Right now it exists both as an artifact in the worktree and as `features.feature_data` rows in SQLite. Which one is authoritative? If the worktree file is canonical, the SQLite rows are a cache and must be rebuilt from the file on every restart. If SQLite is canonical, the worktree file is a render and agents shouldn't be editing it. The current plan is ambiguous and a decision is needed before the artifact-validator and pipeline-engine modules get written.

- **When the review phase fails and loops back to `implement`, does it re-enter with `-c` continuation of the original implementer session or as a fresh session?** §515 (`on_fail: implement`) and §142-146 ("new implementer session with aggregated review notes as context") suggest fresh. §123 retry loop uses `-c`. These are different kinds of failure and need different answers, but the plan uses one word ("retry") for both. Confirm the intended behavior for the review-triggered re-implement case specifically.

- **What is the intended concurrency model for multiple workflows?** §751 says "Named workflows with parallel support (each in its own worktree)." The SQLite schema has no row-level locking concept, better-sqlite3 is synchronous, and the Pipeline Engine is presumably single-threaded. Can two workflows run simultaneously? If yes, are they in the same Node process or separate? This affects whether SQLite writes need transactions beyond the default, and whether the Process Manager needs a spawn queue.

- **Should the harness own `.claude/hooks/` or require the project to own them?** §232 says hooks are "configured in the project's `.claude/hooks` and referenced by jig profiles," implying the project owns them. But `harness init` (§761, §833) is supposed to scaffold a working setup. Does `harness init` write hook scripts into the target repo? If yes, updates to the shipped hooks become harness releases that collide with user edits. If no, the v1 out-of-box experience has a gap the user has to fill before the Stop-hook guarantee holds.

- **Is `architecture.md` a planner output or an input?** §89 lists it as a plan-phase output artifact. But in real projects, the architecture usually exists before planning begins (it's the thing the features must conform to). If the planner is *writing* it, the planner is making architectural decisions without any persistent context — which is unusual and probably wrong for an existing codebase. Is this meant for greenfield only, or should `architecture.md` be an input the user supplies that the planner updates?
